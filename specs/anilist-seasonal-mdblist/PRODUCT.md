# animelistgen

## Summary

A CLI tool and background daemon that fetches all anime for every season
of a given year from AniList and creates or updates corresponding MDBList
lists. Configured via a single YAML file. Designed to run as a one-shot
command, a scheduled systemd timer, or a long-lived daemon with a configurable
sync interval.

## Problem

Agregarr has no built-in "current season anime" source that captures every
show airing that season — only trending/popular subsets. Community-maintained
Trakt and MDBList lists for seasonal anime are unreliable and often abandoned.
This tool replaces manual list curation by generating authoritative lists
directly from AniList's seasonal data and publishing them to MDBList, where
Agregarr can consume them as Custom List sources.

A config-file-driven design ensures the setup is reproducible, portable, and
self-documenting — ideal for a cron job, systemd timer, or container.

## Behavior

### Configuration file

1. The tool looks for a config file at the following paths, in order of
   precedence:
   - Path specified via the `-config` / `-c` CLI flag.
   - `./animelistgen.yaml` in the current working directory.
   - `$XDG_CONFIG_HOME/animelistgen/animelistgen.yaml` (defaults to
     `~/.config/animelistgen/animelistgen.yaml` if `XDG_CONFIG_HOME` is
     unset).

2. The config file is YAML. An example file is bundled with the tool or
   generated via `animelistgen init-config`.

3. The config file has the following top-level structure:

   ```yaml
   # MDBList API key (required). Can also be set via MDBLIST_API_KEY env var.
   mdblist_api_key: ""

   # How often to run in daemon mode. Parsed as Go duration.
   # Default: 24h. Set to 0 for one-shot mode.
   interval: 24h

   # AniList query settings
   anilist:
     # Year to fetch. Default: current year.
     year: 0
     # Seasons to include. Default: all four.
     # Options: winter, spring, summer, fall, or "all".
     seasons: ["all"]
     # Maximum results per season from AniList. Default: 100.
     max_per_season: 100
     # Include ONA format alongside TV. Default: true.
     include_ona: true

   # MDBList list settings
   mdblist:
     # List title template. Supports {season} and {year} placeholders.
     # Default: "Anime {season} {year}"
     title_template: "Anime {season} {year}"
     # List description template. Supports {season} and {year}.
     description_template: >
       All anime TV series airing in {season} {year},
       sourced from AniList. Generated by animelistgen.
     # Set to false to create private lists. Default: true.
     public: true

   # Logging
   logging:
     # One of: debug, info, warn, error. Default: info.
     level: info
     # Path to log file. Empty means stderr.
     file: ""
   ```

4. Unknown keys in the config file cause the tool to print a warning and
   continue — they do not prevent startup. This allows forward-compatible
   config additions.

### CLI modes

5. The tool supports three mutually exclusive modes, selected by the first
   positional argument or a flag:

   | Invocation | Mode | Behavior |
   |---|---|---|
   | `animelistgen` | **oneshot** | Runs once, processes all configured seasons, exits. |
   | `animelistgen daemon` | **daemon** | Runs in background, repeats at `interval`, logs to file. |
   | `animelistgen init-config` | **init** | Writes a default config file to the first config path found or to stdout. |
   | `animelistgen validate` | **validate** | Validates config and API connectivity, then exits. |

6. CLI flags:
   - `-config` / `-c` — path to config file (overrides default search paths).
   - `-dry-run` — runs without making MDBList API calls. Prints what would be
     created.
   - `-output` / `-o` — directory path. Writes JSON files instead of MDBList
     calls (one-shot mode only).
   - `-v` / `-verbose` — more detailed logging.
   - `-h` / `-help` — prints usage and exits.

7. When run without a config file and no config exists at any default path,
   the tool prints an error with instructions to run `animelistgen init-config`
   and exits with code 1.

### Data source: AniList

8. For each configured season, the tool queries the AniList GraphQL API
   (`https://graphql.anilist.co`) for all media matching:
   - `type: ANIME`
   - `season: {WINTER|SPRING|SUMMER|FALL}` (uppercased from config)
   - `seasonYear: {year}`
   - `format_in: [TV]` (and `ONA` if `include_ona` is true)

9. AniList pages results. The tool fetches up to `max_per_season` results
   (one page). If a season has more results, the tool logs a warning at
   debug level but does not paginate further.

10. For each show, the tool extracts:
    - `id` (AniList ID)
    - `title.english` or `title.romaji` (prefer English)
    - `format` (TV or ONA)
    - `episodes` (may be null for currently airing)
    - `genres` (array of strings)
    - `status` (e.g. `RELEASING`, `FINISHED`, `NOT_YET_RELEASED`)
    - `idMal` (MyAnimeList ID, may be null)

11. If the AniList API returns a non-200 response, the tool retries up to 3
    times with exponential backoff (1s, 2s, 4s) before logging an error and
    skipping that season. Other seasons are still processed.

### MDBList output

12. For each season, the tool resolves the list title from the template:
    `Anime {season} {year}` → e.g. "Anime Summer 2026". The season name
    is capitalized (`Winter`, `Spring`, `Summer`, `Fall`).

13. The list description is resolved from `description_template` with the
    same placeholders.

14. Each show is added to the MDBList list using its **display title**
    (English if available, otherwise romaji).

15. Before creating a new list, the tool checks whether a list with the
    matching title already exists on the authenticated MDBList account.
    If it does, the tool **updates** the existing list in-place (replaces
    all items) rather than creating a duplicate.

16. List comparison is title-based (case-insensitive). The tool does not
    delete or modify lists that don't match the configured title pattern.

17. The update is diff-based: the tool compares the current item titles
    against what would be sent. If they match exactly (same set, same
    order), the update is skipped and a debug log entry is emitted.

18. If `public` is `true`, the list is created as public. If `false`, the
    list is created as private.

19. After each MDBList API call, the tool waits at least 1 second before
    the next call. If a 429 response is received, it retries up to 3 times
    with exponential backoff (2s, 4s, 8s) before logging an error and
    skipping that list.

### Output modes

20. **Normal mode** (no `-output`): creates or updates MDBList lists, prints
    the list URL for each season to stdout (one-shot) or logs it (daemon).

21. **Dry-run mode** (`-dry-run`): prints what it would create per season
    (title, show count, changed items). No MDBList API calls are made.

22. **Output-to-file mode** (`-output <dir>`): writes a JSON file per season
    instead of calling MDBList. File names: `anime-{season}-{year}.json`
    (lowercase). Each file contains the full show data from AniList plus a
    timestamp.

### Daemon mode

23. When invoked as `animelistgen daemon`, the tool enters a loop:
    - Runs a full sync cycle (all configured seasons).
    - Sleeps for `interval` duration.
    - Repeats until the process receives SIGINT or SIGTERM, at which point
      it completes the current API call gracefully and exits with code 0.

24. If `interval` is 0 or omitted, daemon mode is rejected with a clear
    error and exit code 1.

25. Daemon mode logs to the configured log file (or stderr if none). Each
    sync cycle logs start, per-season results, and completion.

26. In daemon mode, stdout is not used for output URLs — they are logged
    at info level to the configured log destination.

### Init and validate modes

27. `animelistgen init-config` writes a default config file to the first
    config path in the search order. If `-config` is given, it writes there.
    If no config path is specified and no config directory exists, it creates
    `~/.config/animelistgen/` and writes the default there. The command does
    not overwrite an existing file unless `-force` is passed.

28. `animelistgen validate` loads the config, validates it, tests API
    connectivity to both AniList and MDBList, and prints a summary.
    Exits 0 on success, 1 on failure.

### Exit codes

29. Exit code 0: success (all configured seasons processed, or graceful
    shutdown in daemon mode).

30. Exit code 1: one or more errors occurred (invalid config, API failures).
    Partial results may have been created.

31. Exit code 2: invalid CLI flags.

### Invariants

32. The tool never deletes or modifies existing MDBList lists whose title
    doesn't match the configured template.

33. Re-running the tool is always safe — it is idempotent by design.

34. The tool must handle network failures gracefully: log the error, skip
    the affected season, and continue. It must never produce stale or empty
    MDBList lists silently.

35. Config file changes take effect on the next sync cycle (daemon mode does
    not hot-reload the config).

### Sample systemd service

36. The repository ships a sample systemd unit and timer:

    `deploy/animelistgen.service`:
    ```ini
    [Unit]
    Description=animelistgen — seasonal anime MDBList sync
    After=network-online.target
    Wants=network-online.target

    [Service]
    Type=oneshot
    ExecStart=/usr/local/bin/animelistgen -c /etc/animelistgen/animelistgen.yaml
    User=animelistgen
    Group=animelistgen
    ```

    `deploy/animelistgen.timer`:
    ```ini
    [Unit]
    Description=Run animelistgen daily

    [Timer]
    OnCalendar=daily
    Persistent=true

    [Install]
    WantedBy=timers.target
    ```

    Using `Type=oneshot` + a systemd timer keeps the design simple
    (no daemon mode needed for most users) and takes advantage of
    systemd's built-in scheduling, logging, and failure handling.
    Users who prefer a long-lived daemon can still use the `daemon`
    subcommand.

### Sample Docker deployment

37. The repository ships `Dockerfile` and `docker-compose.yml`:

    ```dockerfile
    FROM golang:1.24-alpine AS builder
    WORKDIR /src
    COPY . .
    RUN go build -o /usr/local/bin/animelistgen ./cmd/animelistgen

    FROM alpine:3.21
    RUN apk add --no-cache ca-certificates tzdata
    COPY --from=builder /usr/local/bin/animelistgen /usr/local/bin/animelistgen
    ENTRYPOINT ["animelistgen"]
    CMD ["daemon"]
    ```

    The `docker-compose.yml` mounts the config file and sets up
    restart policy for daemon mode.
